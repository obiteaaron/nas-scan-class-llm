## 二、用户偏好分析系统

### 2.1 架构概览

```
┌─────────────────────────────────────────────────────────┐
│                    用户交互层                             │
│  搜索 → 点击 → 预览 → 收藏 → 打标 → 访问文件目录          │
└─────────────────────────────────────────────────────────┘
                            ↓ 行为数据收集
┌─────────────────────────────────────────────────────────┐
│                    数据存储层                             │
│  file_views | search_history | user_actions | preferences│
└─────────────────────────────────────────────────────────┘
                            ↓ 偏好分析引擎
┌─────────────────────────────────────────────────────────┐
│                    AI分析层 (可选LLM)                     │
│  偏好画像生成 | 推荐计算 | 分类优化建议                    │
└─────────────────────────────────────────────────────────┘
                            ↓ 结果输出
┌─────────────────────────────────────────────────────────┐
│                    应用层                                 │
│  首页推荐 | 搜索优化 | 分类调整 | 偏好报告卡片              │
└─────────────────────────────────────────────────────────┘
```

### 2.2 数据库设计（新增表）

#### 文件访问记录表 `file_views`

```sql
CREATE TABLE file_views (
    id INTEGER PRIMARY KEY,
    file_id INTEGER NOT NULL,
    view_count INTEGER DEFAULT 0,
    last_viewed_at DATETIME,
    preview_count INTEGER DEFAULT 0,  -- 预览次数
    play_duration INTEGER DEFAULT 0,  -- 播放时长(秒)，针对视频/音频
    FOREIGN KEY (file_id) REFERENCES files(id) ON DELETE CASCADE
);
```

#### 用户操作日志表 `user_actions`

```sql
CREATE TABLE user_actions (
    id INTEGER PRIMARY KEY,
    action_type TEXT NOT NULL,  -- 'view', 'preview', 'play', 'favorite', 'tag', 'search'
    file_id INTEGER,
    tag_id INTEGER,
    search_query TEXT,
    action_data TEXT,           -- JSON格式的额外数据
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP
);
```

#### 用户偏好画像表 `user_preferences`

```sql
CREATE TABLE user_preferences (
    id INTEGER PRIMARY KEY,
    preference_type TEXT NOT NULL,  -- 'category', 'tag', 'keyword', 'time_pattern'
    preference_key TEXT NOT NULL,   -- 具体值，如'视频'、'科幻'
    preference_value REAL,          -- 偏好强度/权重 0-1
    data_source TEXT,               -- 'views', 'favorites', 'tags', 'search'
    last_updated DATETIME,
    UNIQUE(preference_type, preference_key)
);
```

#### 推荐结果缓存表 `recommendations`

```sql
CREATE TABLE recommendations (
    id INTEGER PRIMARY KEY,
    rec_type TEXT NOT NULL,         -- 'similar', 'new', 'trending'
    file_id INTEGER NOT NULL,
    score REAL,                     -- 推荐分数
    reason TEXT,                    -- 推荐理由（AI生成）
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    expires_at DATETIME,            -- 缓存过期时间
    FOREIGN KEY (file_id) REFERENCES files(id) ON DELETE CASCADE
);
```

### 2.3 功能模块设计

#### 模块1：行为数据收集（后端）

**新增API：**
- `POST /api/files/:id/view` - 记录文件查看
- `POST /api/files/:id/preview` - 记录预览（含播放时长）
- 搜索时自动记录点击结果

**配置开关：**
```json
{
  "trackingEnabled": true,
  "trackingLevel": "full"  // 'minimal' | 'full'
}
```

特性：可选开关，用户可决定是否启用行为追踪。

#### 模块2：偏好分析引擎

**统计分析（基础版，无LLM）：**
```
偏好计算公式：
- 类别偏好 = (view_count × 0.3 + favorite_count × 0.5 + tag_count × 0.2) / total_actions
- 标签偏好 = 用户使用该标签频率 × 标签对应文件被访问频率
- 关键词偏好 = 搜索关键词频率 × 搜索结果点击率
```

**LLM增强版（可选）：**
- 输入：用户行为聚合数据
- 输出：偏好画像描述、推荐理由生成
- 示例："你偏爱观看2020年后的科幻电影，对高评分影片关注度较高"

#### 模块3：推荐引擎

**推荐策略：**
1. **相似推荐**：基于用户偏好的类别/标签找相似文件
2. **发现推荐**：用户未看过但符合偏好画像的文件
3. **时间推荐**：根据用户访问时间模式（如周末偏好看长视频）

#### 模块4：智能分类优化

- 分析用户实际使用与当前分类规则的偏差
- 生成优化建议：如"建议将.mkv统一归为'电影'而非'视频'"
- 用户可一键采纳或手动调整

### 2.4 前端集成位置

| 页面 | 增加内容 |
|------|----------|
| **首页** | 「为你推荐」卡片 + 「你的偏好」摘要卡片 |
| **搜索页** | 基于偏好优化搜索排序 + 搜索建议 |
| **文件详情页** | 「相似文件推荐」区块 |
| **设置页** | 「偏好分析」开关 + 「隐私数据管理」 |
| **统计页** | 新增「偏好分析」标签页（可视化偏好分布） |

### 2.5 偏好分析目的

- **个性化推荐**：根据偏好推荐相似文件、新内容
- **偏好报告**：生成用户喜好统计报告，如"你喜欢看科幻电影"
- **智能分类优化**：根据偏好自动调整分类/标签规则

## 三、LLM 具体应用点

| 功能 | LLM任务 | 模型选择建议 |
|------|----------|-------------|
| 偏好画像生成 | 从行为数据生成自然语言描述 | 轻量模型 (如Qwen-7B) |
| 推荐理由生成 | "推荐这部电影因为..." | 可本地/调用API |
| 文件智能分类 | 分析文件名推断分类 | 可批量处理，用轻量模型 |
| 搜索意图理解 | "周星驰的喜剧" → 结构化查询 | 需一定推理能力 |
| 标签智能推荐 | 根据文件名/路径推荐标签 | 可缓存结果 |
| 文件名解析 | `The.Matrix.1999.mkv` → {title: "The Matrix", year: 1999, type: "电影"} | 需正则+LLM结合 |

## 四、实现优先级

| 优先级 | 功能 | 理由 |
|--------|------|------|
| **P0** | 行为数据收集 + 基础偏好计算 | 核心基础，无LLM也可运行 |

| **P3** | 自然语言搜索 + 智能问答 | 体验提升 |

## 六、技术选型建议

### LLM方案

| 方案 | 优点 | 缺点 | 适用场景 |
|------|------|------|----------|
| **本地部署 (Ollama)** | 无网络依赖、隐私安全 | 需GPU资源、响应较慢 | 文件分类、批量处理 |
| **云端API (OpenAI/通义千问)** | 响应快、无需本地资源 | 有成本、需网络 | 搜索意图理解、实时推荐 |
| **混合方案** | 批量任务本地、实时任务云端 | 架构复杂 | 大规模应用 |

### 推荐算法

- **协同过滤**：基于标签/收藏相似度
- **内容过滤**：基于文件属性（类别、大小、年份）
- **混合推荐**：结合用户偏好 + 文件特征

## 七、风险与挑战

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| LLM响应延迟 | 用户等待体验差 | 异步处理 + 缓存结果 |
| 数据隐私 | 用户担忧 | 可选开关 + 数据可清除 |
| 分类准确率 | 用户信任度 | 提供手动调整 + 反馈机制 |
| API成本 | 长期运营压力 | 本地模型 + 结果缓存 |
